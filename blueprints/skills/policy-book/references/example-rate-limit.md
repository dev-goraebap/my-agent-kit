# POL-0017 API 레이트 리밋 정책

## Context
이전 정책(`POL-0008`)은 인증 여부만 기준으로 단일 제한을 걸어 내부 파트너·무료 사용자·유료 사용자가 동일 제한을 공유했다. 2026년 2월 파트너 A가 대량 조회로 전체 서비스 응답이 지연된 사례(인시던트 #INC-2026-021)를 계기로 계층별 레이트 리밋이 필요해졌다.

## Decision
사용자 계층별 분당 요청 제한:

| 계층 | 분당 요청 | 일일 총량 | 버스트 허용 |
|------|-----------|-----------|------------|
| 비인증 | 20 | 1,000 | 30 (10초간) |
| 일반 회원 | 100 | 10,000 | 150 (10초간) |
| 유료 회원 | 600 | 100,000 | 1,000 (10초간) |
| 파트너 API | 3,000 | 제한 없음 | 5,000 (10초간) |

- 제한 초과 시 HTTP 429 응답 + `Retry-After` 헤더
- 제한은 `user_id` 기준 (비인증은 IP 기준)
- 웹훅 수신 엔드포인트는 예외 (§Exceptions)

## Consequences
- **긍정**: 한 사용자의 과다 요청이 전체 서비스에 영향 주는 사태 방지
- **긍정**: 유료 등급의 실질 혜택 명확화 (전환율 상승 기대)
- **부정**: 비인증 제한이 공격적이라 초기 사용자 이탈 가능성
- **부정**: Redis 기반 카운터 운영 비용 증가 (월 약 15% 상향 추정)

## Exceptions
- **웹훅 수신 엔드포인트** (`POST /webhooks/*`): 제한 없음 (외부 시스템이 제어 불가)
- **헬스체크** (`GET /health`): 제한 없음
- **관리자 호출** (`role: admin`): 유료 회원 제한 × 5

## 처리 절차
1. API Gateway에서 `user_id` 추출 (Identity 도메인 세션 조회)
2. Redis의 `rate_limit:{user_id}:{window}` 키 증가
3. 초과 시 429 반환 + `Retry-After: <seconds>` 헤더
4. 일일 총량 초과는 자정(KST)에 리셋
5. 파트너 API는 별도 키 네임스페이스로 격리

## 근거
- 인시던트 #INC-2026-021 (2026-02-18, 파트너 A 과다 조회로 p95 응답시간 3초→28초)
- 2026-03 플랫폼 팀 RFC `RFC-PLT-003 — Tiered Rate Limiting`
- 경쟁사 벤치마크 (Stripe, Twilio 공개 가이드 참조)

## 변경 이력
- 2026-03-15: POL-0008 대체, 계층별 세분화 도입 (effective)
- 2026-03-10: 파트너 A 인시던트로 POL-0008 한계 노출, 재설계 착수
