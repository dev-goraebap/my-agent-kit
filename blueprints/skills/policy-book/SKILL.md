# policy-book 스킬

정책은 "**비즈니스 의사결정으로 바뀔 수 있는 규칙**"이다. 도메인 불변 규칙("주문은 최소 1개 항목")과는 달리 값·기간·조건이 바뀌어도 도메인은 멀쩡하다. 이 스킬은 그런 정책을 법전처럼 한 곳에 모아 **번호와 이력**으로 추적한다.

## 레퍼런스

`references/`에 완성 예제 2종이 있다.

| 예제 | 파일 | 특징 |
|------|------|------|
| 환불 정책 | `references/example-refund.md` | 전형적 B2C, 법적 근거 + 예외 |
| API 레이트 리밋 | `references/example-rate-limit.md` | 운영 정책, 계층별 수치 |

---

## 1. 언제 트리거하는가

- "환불 정책 만들어줘", "배송 정책 수립", "쿠폰 사용 규칙 정의"
- "레이트 리밋 정책", "비밀번호 정책"
- "POL-0005 업데이트", "이 정책 개정해줘"
- "/policy-book", "정책 법전에 추가"
- "이거 정책으로 정리해줘"

사용자가 도메인 모델 작성 중 "이건 정책에 넣는 게 맞겠다"고 판단하면 `domain-models` 스킬이 이 스킬로 위임할 수도 있다 (수동 전환).

---

## 2. 저장 경로 규약

`domain-models`과 동일 규약이다. **별도 config 파일도, 전용 섹션도 만들지 않는다.** 루트 `AGENTS.md`(우선) 또는 `CLAUDE.md`의 **기존 `## References` 섹션**에 한 줄로 얹는다.

### 2-1. 최초 실행 절차

1. source 파일을 선택한다 — `AGENTS.md` → `CLAUDE.md` 순.
2. `## References`(또는 등가) 섹션에서 **정책 폴더 경로를 찾는다** — "Policy book:", "Policies:", "정책 폴더:" 같이 이 스킬 용도로 해석되는 한 줄.
3. 찾으면 그 경로 사용 → 3-2로 진행.
4. 못 찾으면 인터뷰:

    ```
    정책 문서를 어디에 저장할까요?
      기본값: docs/policies/
      (폴더명을 바꾸려면 예: 'rules', 'policies-book', 'codex')

    입력:
    ```

5. source 파일의 `## References`에 한 줄 추가(섹션이 없으면 문서 끝에 신설):

    ```markdown
    ## References
    ...
    - Policy book: `docs/policies/`
    ```

6. 이후 재질문 X.

### 2-2. 정책 폴더 신규 생성 시

폴더를 처음 만들면 `{policies_folder}/index.md`도 함께 생성한다 (인덱스는 §7 참조). 이후 정책을 추가하면 인덱스 자동 갱신.

## 3. 정책 파일 구조

### 3-1. 파일명 규칙

```
POL-{4자리 ID}-{slug}.md
```

예: `POL-0001-refund.md`, `POL-0002-cancel-order.md`, `POL-0017-rate-limit.md`

- ID는 4자리 고정 (`POL-0001` ~ `POL-9999`). 9999개까지 확장 없이 커버.
- slug는 영문 소문자 + 하이픈. 한국어 금지 (파일명 이식성).

### 3-2. frontmatter

```yaml
---
id: POL-0001
title: 환불 정책
status: active           # proposed | active | superseded | deprecated
effective_from: 2026-01-01
effective_to: null       # null = 기한 없음, 날짜면 그날까지만 유효
applies_to:              # 적용 대상 도메인 slug (여러 개 가능)
  - order
  - payment
supersedes: null         # POL-xxxx — 이 정책이 대체한 이전 정책
superseded_by: null      # POL-xxxx — 이 정책을 대체한 다음 정책
owner: pm-team
---
```

모든 필드는 정책 추적에 의미가 있다. **`applies_to`는 `domain-models`과의 양방향 링크 키**다 — 정책 생성 시 각 도메인 DOMAIN.md의 `Related Policies` 섹션에 링크 추가 제안.

### 3-3. 섹션 구조

#### 필수 3섹션 (ADR 스타일)

| # | 섹션 | 용도 | 분량 |
|---|------|------|------|
| 1 | **Context** | 왜 이 정책이 필요한가 — 배경·pain·상위 요구 | 2~4문장 |
| 2 | **Decision** | 무엇을 정했나 — 조건·수치·기간 | 불릿 또는 표 |
| 3 | **Consequences** | 이 결정의 결과 — 긍정/부정 영향 | 불릿 2~4개 |

#### 선택 섹션

| 섹션 | 언제 추가 |
|------|-----------|
| **Exceptions** | 예외 조건이 있을 때 |
| **처리 절차** | 구현자가 참고할 단계가 있을 때 |
| **근거** | 법적·규제·내부 승인 문서 링크 |
| **변경 이력** | 정책 값이 한 번이라도 바뀌면 반드시 추가 (§4-1) |

## 4. 라이프사이클 관리 — **덮어쓰지 않는다**

정책은 법전이므로 과거 버전도 남긴다. 세 가지 시나리오:

### 4-1. 값만 변경 (같은 정책의 업데이트)

예: "반품 기간 7일 → 14일"

처리:
- 같은 파일을 수정한다.
- **`변경 이력` 섹션(없으면 생성)에 엔트리 추가**:

```markdown
## 변경 이력
- 2026-04-19: 반품 기간 7→14일로 확대 (CS 이탈률 대응)
- 2025-07-01: 최초 수립
```

- `effective_from` 갱신 (이번 변경 적용일).
- `status`는 `active` 유지.

### 4-2. 정책의 완전한 대체 (성격이 바뀜)

예: "환불 → 교환만 허용으로 바꿈"

처리:
- **새 파일 `POL-XXXX` 생성** (다음 ID 채번).
- 새 파일의 frontmatter에 `supersedes: POL-0001`.
- **기존 파일(`POL-0001`) 수정**:
  - `status: superseded`
  - `superseded_by: POL-00XX`
  - `effective_to: <새 정책 발효일>`
- 기존 파일은 **삭제하지 않는다**. 법전은 과거 조항도 보존.

### 4-3. 정책 폐지 (대체 없이 그냥 없앰)

예: "코로나 기간 한정 환불 연장 정책" 종료

처리:
- `status: deprecated`
- `effective_to: <폐지일>`
- `변경 이력`에 폐지 이유 기록
- 파일은 유지

## 5. 인터뷰 절차

### 5-1. 새 정책 생성

```
어떤 정책을 만들까요?
  제목: (예: "환불 정책", "API 레이트 리밋")
  적용 도메인: (예: order, payment — 여러 개 가능)
```

→ ID 자동 채번 제시 (`POL-0017 (다음 번호)` 같이 표기) → 사용자 확인.

이어서 **ADR 3섹션**을 차례로:

```
(1) Context — 왜 이 정책이 필요한가요? 2~3문장으로.

(2) Decision — 무엇을 정하셨나요? 조건·수치·기간.

(3) Consequences — 이 결정이 가져올 결과(긍정/부정)는?
```

선택 섹션 포함 여부는 마지막에 한 번에:

```
추가 섹션이 필요한가요?
  [ ] Exceptions (예외 조건)
  [ ] 처리 절차 (구현 안내)
  [ ] 근거 (법적/내부 문서 링크)
```

### 5-2. 기존 정책 업데이트

```
어떤 정책을 수정하나요?
  POL-ID: POL-0001
  변경 성격:
    (a) 값만 변경 (같은 정책 내 업데이트) → §4-1
    (b) 정책을 완전히 대체 → §4-2 (새 POL 생성)
    (c) 정책 폐지 → §4-3
```

각 경로에 맞는 절차 수행.

### 5-3. 기존 정책 조회

```
현재 정책 목록을 보여줄까요? (y/n)
  → status별 그룹핑 (active / superseded / deprecated) + 제목 목록
```

---

## 6. `applies_to` 양방향 링크

정책을 생성·수정할 때 `applies_to`에 적힌 각 도메인의 `DOMAIN.md`를 확인해 `Related Policies` 섹션에 링크를 추가 제안한다.

```
POL-0005를 order 도메인에 적용하셨네요.
docs/domains/order/DOMAIN.md의 Related Policies 섹션에 링크를 추가할까요? (y/n)
```

- **추가는 사용자 동의 후.** 자동으로 도메인 문서를 수정하지 않음 (Ask first 성격).
- 도메인이 존재하지 않으면 "이 도메인은 아직 문서가 없어요. `domain-models` 스킬로 먼저 만드시겠어요?" 제안.

마찬가지로 정책의 `applies_to`에서 도메인을 제거하면 해당 DOMAIN.md의 링크 제거도 제안.

---

## 7. 인덱스 관리 — `index.md`

`{policies_folder}/index.md`는 전체 정책의 목차다. 정책 추가·상태 변경 시 스킬이 자동 갱신한다.

### 구조

```markdown
# Policies

## Active
- [POL-0001 환불 정책](POL-0001-refund.md) — order, payment
- [POL-0003 배송비 정책](POL-0003-shipping-fee.md) — order

## Superseded
- [POL-0002 주문 취소 정책 v1](POL-0002-cancel-v1.md) → POL-0017

## Deprecated
- [POL-0004 COVID 환불 연장 특례](POL-0004-covid-refund.md) — 2022-06-30 폐지
```

- `applies_to` 값을 정책 제목 옆에 태그처럼 표시 → 도메인별로 훑기 쉬움.
- 검색성을 위해 파일명 기반 링크 유지.

## 8. 작성 규칙

### 8-1. Context는 **정책 없던 시절의 문제**를 쓴다

"CS 문의가 월 200건 발생했다. 그중 환불 관련이 30%였다"처럼 **구체적 숫자·사건**이 좋다. 막연한 서술("사용자 편의를 위해")은 지양.

### 8-2. Decision은 **검증 가능한 형태**로

- O: "주문 후 7일 이내 환불 허용. VIP는 14일."
- X: "합리적인 기간 내 환불 처리"

### 8-3. Consequences는 양면 모두 적는다

긍정만 적으면 정책 자체 정당화가 되어버린다. 부작용도 한 줄이라도 적는다.

### 8-4. `근거`는 링크로 남긴다

- 법적 근거: 조문 번호 명시 (`전자상거래법 제17조`)
- 내부 문서: 상대경로 링크
- 회의록: 링크가 있으면 첨부, 없으면 "2026-01 기획 회의 결정" 수준의 출처

### 8-5. 정책 수정 시 `변경 이력`은 반드시 갱신

`status`나 `effective_from` 건드리면 자동으로 `변경 이력` 추가 제안.

---

## 9. 생성 → 리뷰 → 개선 워크플로우

1. **새 정책** — §5-1 인터뷰 → 파일 생성 + 인덱스 갱신 + DOMAIN.md 링크 제안.
2. **정책 수정** — §5-2에서 변경 성격 확인 후 §4-1/4-2/4-3 분기.
3. **정책 조회** — §5-3 또는 인덱스 파일 참조.
4. **정합성 점검** — 정기적으로 "superseded인데 `superseded_by` 비어 있음" 같은 일관성 문제를 리포트할 수 있다 (후속 동작, 지금은 범위 밖).
