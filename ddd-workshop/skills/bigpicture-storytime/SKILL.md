# bigpicture-storytime 스킬

ddd-workshop 파이프라인의 **첫 스킬**. **빅픽처 이벤트스토밍의 1:1 학습 친화 변형**이다.

전통적인 빅픽처 이벤트스토밍은 도메인 전문가들이 큰 벽 앞에 모여 시간순 이벤트를 던지는 그룹 워크샵이다. 그런데 SI·신규 프로젝트 환경에선 도메인 전문가에 즉시 접근하기 어렵고, 개인이 도메인을 빠르게 이해해야 할 때가 많다. 이 스킬은 그런 1:1 환경에 맞춰 빅픽처를 다시 짠다.

**핵심 차이는 "결과물"이 아니라 "만들어지는 과정"**:

| | 빅픽처 이벤트스토밍 | bigpicture-storytime |
|------|-------------------|---------------------|
| 산출물 | 시간순 이벤트·액터·외부시스템·핫스팟·피벗 | 동일 |
| 만들어지는 방식 | 한 보드에 한 번에 펼침 (Chaotic exploration) | 한 액터·한 챕터씩 시간순 서사로 누적 |
| 시점 | 여러 액터 동시 | 한 액터 시점 우선, 다음 액터로 전환 |
| 진행자 | 그룹 + 도메인 전문가 + 퍼실리테이터 | 사용자(학습자) + AI(가상 도메인 전문가) |
| 인지 부담 | 한 번에 모두 봐야 함 | 한 챕터씩 처리 |

핵심 철학: **이해가 산출물보다 먼저다.** 이 스킬을 돌린 결과로 문서가 안 남아도 괜찮다. 사용자 머릿속에 빅픽처가 들어갔으면 성공이다.

기법 계보: Stefan Hofer / Henning Schwentner의 **Domain Storytelling**을 베이스로 하되, AI 가상 도메인 전문가 환경에 맞게 학습 친화적으로 변형했다. 원본 Domain Storytelling에 없는 추가 요소: 각 장면마다 "왜 이게 필요한가?" 설명, 확신도 태그, 챕터 단위 인터랙션.

---

## 1. 언제 트리거하는가

- `/ddd-workshop:bigpicture-storytime`
- 사용자가 처음 보는 도메인을 다루어야 할 때
- "이 도메인을 잘 모르겠다", "차근차근 이해하고 싶다"
- RFP·요구사항정의서를 받았는데 비즈니스 흐름이 잘 안 그려질 때
- 신규 SI 프로젝트 킥오프, 클라이언트 미팅 전 도메인 빌드업
- 보편적 도메인 패턴을 빠르게 파악하고 싶을 때 (배달, 예약, 학사 등)

이 스킬 후 다음 단계로 갈지는 사용자 판단. ddd-workshop의 다른 스킬로 자연스럽게 이어가도 되고, 이 스킬만 쓰고 멈춰도 된다.

---

## 2. 핵심 컨셉 3가지

### 2.1 이해 우선, 산출물은 부산물

기존 워크샵 산출물이 관리 안 되는 핵심 원인은 "**산출물부터 만들고 이해는 그 다음**"이라는 역순 진행이다. 이 스킬은 그 순서를 뒤집는다.

- **저장은 옵셔널**: 사용자가 명시 거부하면 대화로만 진행
- **기본 경로** `docs/bigpicture.md` — 사용자가 언제든 변경 가능
- **자동 누적**: 기본 경로 또는 지정 경로에 파일이 이미 있으면 이어서 누적 (Step 8 참조)
- **진화 친화적**: 한 번 만들고 끝이 아니라 챕터마다 자라는 형태. 다른 스킬·인터뷰의 1차 입력으로 쓸 수 있는 구조

### 2.2 액터별·시간순 서사

**한 명의 페르소나** 시점에서 **시간순으로** 도메인을 풀어낸다. 한 액터의 여정을 끝까지 따라간 뒤, 다른 액터로 시점을 옮겨 같은 도메인을 다시 본다. 이 누적이 도메인 전체 그림을 만든다.

이벤트 스토밍과의 차이: 이벤트 스토밍은 시스템 전체 이벤트를 한꺼번에 던진다. 이 스킬은 한 사람의 이야기를 따라간다. 인지 부담이 훨씬 적다.

### 2.3 확신도 태그로 검증 지점 명시

AI는 도메인 일반 패턴은 알지만 특정 클라이언트의 현장 사정은 모른다. 그래서 모든 정보에 확신도를 표시한다.

- `[확실]` — 산업 표준·법령에 명문화된 것
- `[일반론]` — 흔한 패턴이지만 회사·현장마다 다를 수 있음
- `[추측]` — AI가 메우는 가설. 현장 검증 필요

이 태그가 "**현장 담당자에게 무엇을 물어볼지**"를 자동으로 표시한다. 세션 종료 후 사용자가 검증할 지점이 명확해진다.

---

## 3. 입력

- **있으면**: RFP, 요구사항정의서, PRD, 기존 도메인 자료
- **없으면**: 사용자가 제공하는 도메인 키워드 (한 줄도 가능)
- **자동 보충**: AI 사전 리서치 (보편 사례·법령·산업 표준·주요 액터)

부분 입력 허용. 자료가 부족하면 사전 리서치로 메운 뒤 진행한다.

---

## 4. 진행 절차

### Step 1 — 사전 리서치 & 도메인 윤곽

사용자 입력을 받은 직후, 짧은 사전 리서치를 수행해 다음을 정리한다:

- 도메인의 큰 페이즈 (시간축 후보)
- 주요 액터 목록
- 외부 시스템 (PG, 인증 등)
- 관련 법령·규정 (있으면)
- 도메인 특유 포인트
- AI가 보기에 의심스러운 핫스팟 (미리 발견된 의문점)

**모든 항목에 확신도 태그를 붙인다.** 사용자 자료가 충분하면 그 기반, 부족하면 보편 사례 리서치 (필요 시 WebSearch).

산출 형식: 짧은 요약 (한 화면 분량). 사용자 확인 받고 다음 단계.

### Step 2 — 페르소나 설정

도메인의 핵심 액터 중 하나를 선택해 **구체적인 페르소나**를 만든다.

- 이름·나이·배경·동기·제약을 짧게 (3~5문장)
- 사용자가 자신을 모델로 하고 싶다고 하면 사용자 정보를 반영
- 이름은 사용자가 별명·닉을 주면 그것 사용

페르소나가 구체적일수록 이야기가 살아난다. "어떤 학생이 ..." 보다 "31세 직장인 김철수가 ..."가 훨씬 강력.

### Step 3 — 액터 인터뷰 흐름 결정 (하이브리드)

여러 액터 중 어디서 시작할지 정한다. 기본은 **하이브리드 방식**:

```
이 도메인의 주요 액터:
1. {액터1} — {역할}
2. {액터2} — {역할}
3. {액터3} — {역할}
...

추천: {액터1}부터 시작합니다.
이유: 이 도메인의 1차 사용자이고 대부분의 흐름이 여기서 시작.

다른 액터로 시작하시겠어요? 아니면 추천대로?
```

### Step 4 — 챕터 단위 스토리텔링

한 챕터 = 도메인의 한 페이즈 (예: "지원과 입학", "수강과 학업").
한 챕터 안에 **5~7개 장면**.

각 장면 구조:

```markdown
### 장면 N. {장면 제목}

> {페르소나 시점의 1~3문장 서사}

**🤔 시스템 관점에서 무엇이 필요할까?**
- {필요 1}
- {필요 2}

**왜 이게 필요한가?**
{1~2문장 설명. 추상이 아니라 구체적 이유}

🟪 **핫스팟 (있으면)**: {의문점·가정·갈등}
```

**한 응답에 한 챕터까지만.** 더 진행할지 사용자에게 묻고 다음 챕터로 넘어간다. 이게 인지 과부하를 막는 가장 중요한 규칙.

### Step 5 — 챕터 정리

챕터 끝에 다음을 표·목록 형태로 정리한다:

- 도출된 도메인 이벤트 (시간순)
- 등장 액터, 외부 시스템
- 누적된 핫스팟

표기 약속:

```
🟧 [E] 도메인 이벤트 (과거형, 예: "회원가입 완료")
🟪 [?] 핫스팟·질문·가정
🟥 [X] 외부 시스템
🟨 [A] 액터
⭐ [P] 피벗 이벤트 (단계가 바뀌는 굵직한 이벤트)
```

#### 산출물 갱신 확인 (자동 누적 모드)

챕터 정리 직후, 산출물 파일 갱신 여부를 사용자에게 확인한다:

- **기본 경로**: `docs/bigpicture.md` — 사용자가 다른 경로로 변경 가능
- **자동 누적**: 기본 경로 또는 지정 경로에 파일이 이미 있으면 별도 확인 없이 **이어서 누적** (`--update` 모드 자동 진입). 새로 시작은 사용자가 명시 요청한 경우에만
- **저장 거부 가능**: 사용자가 "저장 안 함"을 선택하면 대화로만 진행
- **확인 한 줄**: "이번 챕터 내용을 `docs/bigpicture.md`에 갱신할까요? (y / 다른 경로 / 저장 안 함)"

저장 시 7개 섹션 통합본을 갱신하고 챕터 아카이브에 이번 챕터를 추가한다 (구체 형식은 Step 8 참조).

### Step 6 — 사용자 인터랙션

챕터 정리 후 다음을 묻는다:

1. **빠진 게 있나?** (사용자 경험·관찰 기준)
2. **"이건 정말 그래?"** 의심점 (AI 일반론 검증)
3. **새 핫스팟** 추가
4. **"이건 너무 디테일한데?"** 빅픽처 단계에서 빼야 할 것
5. **다음 챕터로 갈지, 멈출지**

### Step 7 — 액터 전환

한 액터의 여정이 끝나면 다음 액터로 시점 전환을 권유한다. 사용자가 "충분하다"고 하면 거기서 멈춘다.

### Step 8 — 세션 종료와 최종 산출물

세션을 명시적으로 종료할 때 (사용자가 "충분하다", "여기까지" 등) 또는 처음부터 산출물을 남길 의사가 있을 때 사용한다.

#### 8.1 산출물 파일 정책

| 항목 | 정책 |
|------|------|
| 기본 경로 | `docs/bigpicture.md` |
| 경로 변경 | 사용자가 언제든 다른 경로 지정 가능 |
| 기존 파일 발견 | **자동 누적** (`--update` 모드 자동 진입). 새로 시작은 명시 요청 시만 |
| 저장 시점 | 챕터 끝마다 사용자 확인 후 갱신 (Step 5 산출물 갱신 확인) + 세션 종료 시 최종 점검 |
| 저장 거부 | 옵셔널. 사용자가 명시 거부 시 파일 없이 대화로만 진행 |

#### 8.2 산출물 구조

단일 마크다운 파일. 위쪽은 **누적 통합본** (다음 스킬·인터뷰의 1차 입력), 아래쪽은 **챕터 아카이브** (발생 순서 기록).

```markdown
# {도메인명} — Big Picture

## 1. 도메인 윤곽
(Step 1 산출 — 페이즈, 액터, 외부 시스템, 법령, 특수성)

## 2. 시간순 이벤트 (통합)
| # | 이벤트 | 액터 | 챕터 |
|---|--------|------|------|
| 1 | 🟧 ... | ... | Ch.1 |

## 3. ⭐ 피벗 이벤트
- ⭐ ... — {왜 피벗인가}

## 4. 🟨 액터 매트릭스
| 액터 | 시스템 계정 | 주요 활동 | 등장 챕터 |

## 5. 🟥 외부 시스템
| 시스템 | 용도 | 의존 강도 |

## 6. 🟪 핫스팟 (검증 필요)
| 항목 | 출처 챕터 | 확신도 |

## 7. 서브도메인·BC 후보 (선택)
누적이 충분해지면 채움. 처음엔 비워둠.

---

## 챕터 아카이브 (참고)

### 챕터 1 — {제목}
(스토리 + 이벤트 + 핫스팟 — 발생한 그대로)

### 챕터 2 — ...
```

핵심: **위 7개 섹션은 누적되며 자란다.** 챕터가 늘어나도 위쪽 통합본이 1차 참고 영역. 아래 챕터 아카이브는 출처 추적·재학습용.

#### 8.3 시각화 (선택)

기본은 텍스트만. 사용자가 "다이어그램 추가해줘", "mermaid 그려줘" 같이 **명시 요청**하면 추가한다. 강제로 항상 그리지 않는다 — 산출물이 무거워지지 않게.

옵션 예시:
- `timeline` — 시간순 이벤트 흐름 시각화
- `flowchart` — 액터 ↔ 외부 시스템 관계도
- `journey` — 페르소나 시점의 단계별 경험 (Domain Storytelling 보강)

#### 8.4 세션 종료 체크리스트

세션이 끝날 때 다음을 점검한다:

1. ✅ 산출물 파일이 최신 상태인가? (마지막 챕터까지 반영)
2. ⚠️ 우선순위 높은 핫스팟 강조 — 현장 검증이 시급한 [추측] 항목들
3. 🔁 다음 세션 가이드 — 다른 액터·다른 페이즈로 이어서 굴릴 수 있음
4. ➡️ ddd-workshop 다음 단계 스킬로 연결 (선택)

---

## 5. Anti-patterns (절대 하지 말 것)

- **한 번에 모든 이벤트 던지기** — 인지 과부하. 챕터당 5~7개 장면 엄수
- **시간 흐름 무시한 Chaotic exploration** — Brandolini의 빅픽처 이벤트스토밍은 도메인 전문가가 같은 공간에 있을 때 통한다. 이 스킬 환경(채팅, AI 가상 전문가)에선 작동 안 함
- **"왜?" 설명 없이 이벤트만 나열** — 학습자가 따라갈 수 없음
- **확신도 표시 누락** — AI가 일반론으로 도망갈 위험
- **챕터 갱신을 사용자 확인 없이** — 자동 누적은 모드 정책일 뿐. 매 챕터 끝에 "갱신할까요?" 한 줄 확인은 꼭 받기 (Step 5)
- **산업 표준 그대로 들이밀기** — 클라이언트 특수성을 묻기 위한 핫스팟으로 변환해야 함
- **액터 전환을 사용자 동의 없이** — 한 액터 끝났다고 곧바로 다음 액터로 가지 않음
- **챕터를 한 응답에 여러 개 욱여넣기** — 한 응답 = 한 챕터

---

## 6. 이 스킬이 끝난 뒤

- 사용자가 도메인을 이해했다 — **이게 1차 성공.**
- (옵션) 정리된 마크다운이 남았다 — 다른 스킬·클라이언트 인터뷰의 입력으로 활용
- (옵션) ddd-workshop의 다음 단계 스킬로 자연스럽게 연결

이 스킬은 **반복 사용 가능**하다. 같은 도메인이라도 새 액터 시점으로, 또는 새 페이즈에 집중해서 다시 돌릴 수 있다.

---

## 7. 짧은 예시 흐름 (방송통신대학원 학사정보 시스템)

```
입력: "방송통신대학원 학사정보 시스템을 이해하고 싶어요"

→ Step 1: 사전 리서치
- 페이즈: 입학 → 학적 → 수강·학업 → 자격검증 → 논문 → 학위
- 액터: 대학원생, 지도교수, 행정실 직원, 심사위원 [확실]
- 외부 시스템: PG사, 본인인증, 표절검사 [확실]
- 특수성: 직장인 위주, 온라인 중심, 면접 비중 낮음 [일반론]

→ Step 2: 페르소나
"고래밥, 31세, 방통대 학부 출신 SI 개발자, 직장 다니면서 진학"

→ Step 3: 액터 선택
추천: 대학원생부터 → 사용자 확인 → 진행

→ Step 4: 챕터 1 — 지원과 입학
장면 1. 모집요강을 본다 (+ 왜? + 핫스팟)
장면 2. 회원가입과 본인 인증
장면 3. 원서 작성
장면 4. 원서 제출과 전형료 결제 (피벗 이벤트)

→ Step 5: 챕터 정리 — 이벤트 8개, 액터 2명, 핫스팟 4개
   → "docs/bigpicture.md에 갱신할까요?" → y → 통합본 누적

→ Step 6: 인터랙션
사용자: "직장인 우대 전형이 있으면 별도로 처리?"
→ 핫스팟 추가, 다음 챕터로 결정

→ ... (반복: 챕터마다 갱신 확인 → 자동 누적)

→ Step 7: 액터 전환 — "이번엔 행정실 직원 시점에서?"

→ Step 8: 세션 종료 — 산출물 최종 점검, 우선순위 핫스팟 강조, 다음 단계 가이드
```

## 8. 한 줄로 정리
