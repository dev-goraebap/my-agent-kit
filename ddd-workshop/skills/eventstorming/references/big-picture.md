# Big Picture 사이클 (references)

> 부모: [`../SKILL.md`](../SKILL.md). 이 파일은 Big Picture Initial / 페이즈 누적 / Cycle 모드 진입 시 추가 로드.

## 1. Big Picture 사이클 정체

빅픽처는 도메인 시간축의 *탐색(divergent)* 작업. 페이즈·도메인 이벤트·액터·외부시스템·핫스팟·Pivotal·Subdomain 후보를 누적.

| 빅픽처에 포함 | 빅픽처에 포함 X |
|---|---|
| 🟧 시간순 도메인 이벤트 | 🟦 Command (Process 영역) |
| ⭐ Pivotal Event | 🟪 Policy (Process 영역) |
| 🟨 Actor | 🟩 Read Model (Process 영역) |
| 🟥 External System | 🟫 Aggregate 정식 정의 (코드 영역) |
| 🔥 Hotspot | BC 확정 (BC Discovery 사이클) |
| **(부록) Subdomain 후보** | UL 정식 용어집 (코드 영역) |

## 2. Step 1 — 모드 진입

| 진입 신호 | 동작 |
|---|---|
| `docs/eventstorming/index.md` 없음 | **Initial** → Step 2 (도메인 윤곽) |
| index.md 있음 + 페이즈 부족 | **페이즈 누적** → Step 4 (페이즈 단위) |
| 시간 지난 후 다시 호출 | **Cycle** → 기존 읽고 사용자에게 변화 발견 묻기 |

## 3. Step 2 — 도메인 윤곽 매핑 (Initial 전용)

다음을 수집·정리 (도메인 학습 노트 + 클라이언트 자료 + 사용자 발화 기반):

- **페이즈 후보** — 시간축 분할
- **다루는 페이즈 vs 다루지 않는 페이즈** — 명시적 범위 (사용자 확인)
- **주요 액터** — 보편 + 클라이언트 특수
- **외부 시스템** — 의존 강도 표
- **관련 법령·표준**
- **도메인 특수성** — 일반 패턴과 다른 점 (호호컴퍼니의 NotificationPolicy listen 패턴 같은 것)

사용자 확인 후 `index.md` 1·2 섹션 작성.

## 4. Step 3 — 페이즈 단위 누적

페이즈 하나씩 다음을 채운다 (한 응답 = 한 페이즈):

```markdown
### 페이즈 N: {페이즈명}

**한 줄 핵심**: {이 페이즈에서 일어나는 일}

**시간순 이벤트** (4기준 통과한 것만):
| # | 이벤트 (한국어, 과거형) | Code Identifier | 액터 | 비고 |
|---|----------------------|----------------|------|------|
| 1 | ⭐ 사원 사전 등록됨 | `EmployeePreRegistered` | 인사담당자 | 페이즈 시작점 |
| 2 | 🟧 ... | `...` | ... | |

**페이즈 핫스팟** (페이즈 한정):
- 🔥 H1-A [추측: 노무] {핫스팟}
- 🔥 H1-B [확실] {핫스팟}

(선택) **절 분리** — 영속 흐름 vs 1회성 백필 같은 별도 흐름이면:
**절 A. {흐름 이름}**: ...
**절 B. {흐름 이름}**: ...
```

핫스팟 태깅 규약:
- ID: `H{페이즈번호}-{알파벳}` (페이즈 1의 첫 핫스팟 = H1-A)
- 확신도: `[확실]` / `[추측]` / `[가설]`
- 답할 위치: `[노무: ...]`, `[운영: ...]`, `[시스템: ...]`, `[법령: ...]`

## 5. Step 4 — 페이즈 검증 인터랙션

페이즈 끝에 사용자에게:

1. **빠진 이벤트?** (사용자 경험·관찰 기준)
2. **이게 정말 도메인 이벤트인가?** (UI/기술 이벤트 의심)
3. **새 핫스팟?**
4. 다음 페이즈 / 더 깊게 / 멈추기

답 안 해도 다음 진행 가능.

## 6. Step 5 — 횡단 정리 (페이즈 2개 이상 누적되면)

`index.md`의 횡단 매트릭스 갱신:
- **🟨 액터 매트릭스** — 액터 → 등장 페이즈
- **🟥 외부 시스템 매트릭스** — 시스템 → 의존 강도 + 등장 페이즈
- **🔥 누적 핫스팟** — 전체 정렬 (ID·페이즈·답할 위치·확신도)
- **(부록 A) UL 메모** — 이벤트·액터에서 도출된 핵심 용어
- **(부록 B) Subdomain 후보** — 페이즈 클러스터링으로 자연 도출 (Khononov heuristic 수준 — Core/Supporting/Generic 분류는 BC Discovery 사이클에서)

## 7. Step 6 — 산출물 갱신 정책

각 페이즈 끝에 사용자 확인:

> "이 페이즈를 `docs/eventstorming/{NN-slug}/bigpicture.md`에 갱신하고 `index.md` 페이즈 맵·횡단 매트릭스를 함께 반영할까요? (y / 다른 경로 / 저장 안 함)"

자동 누적 — 폴더·파일 있으면 변경분 추가, 없으면 신규 생성.

## 8. Step 7 — Cycle 모드 (코드 작성 후 사이클)

코드 작성 중 빅픽처 차원 변경(새 페이즈·BC 후보 변동·새 횡단 핫스팟) 발견 시:

1. 기존 `docs/eventstorming/` 전체 읽기
2. 사용자에게 "어떤 변화가 있었나요?" 묻기
   - 새 이벤트?
   - 페이즈 분리·통합?
   - 핫스팟 해소·새 발견?
3. 변경분만 갱신
4. 변경 이력 자동 메모 (선택)

## 10. Anti-patterns (Big Picture 한정)

- **페르소나 시점 서사** — 학습 톤
- **챕터·장면 단위** — 페이즈 단위
- **UI/Telemetry/기술 이벤트 혼입** — 4기준 게이트
- **Command·Policy·Read Model 추가** — Process Modeling 사이클
- **BC 확정** — BC Discovery 사이클
- **Subdomain Core/Supporting/Generic 분류** — BC Discovery 사이클 (여기는 *후보 식별*만)
- **페이즈 한 응답에 여러 개 욱여넣기** — 한 응답 = 한 페이즈
