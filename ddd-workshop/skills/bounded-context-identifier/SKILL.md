# bounded-context-identifier 스킬

**Bounded Context(BC)** 는 DDD 전략적 설계의 심장이다. 한 모델이 유효한 경계를 그어, 그 안에서는 용어·규칙·책임이 일관되게 쓰이도록 한다. 잘못 그으면 이후 Aggregate 설계가 통째로 어긋난다.

이 스킬은 앞 스킬들이 모아둔 증거를 **가장 강력한 순서로** 조립해 BC 후보를 제안한다.

## 3. 판별 기준 (우선순위 순)

### 1순위: 같은 용어, 다른 의미 (동음이의어)
용어집에서 같은 단어가 다른 속성 집합을 갖는 지점은 **가장 강력한 BC 분리 증거**. UL 스킬 출력의 "동음이의어 후보" 섹션을 가장 먼저 본다.

### 2순위: 응집도
자주 함께 변하는 개념들은 한 BC. `event-flow-mapper`의 정책·이벤트 클러스터를 살핀다.

### 3순위: 책임·조직 (Conway's Law)
다른 팀이 관리하거나 다른 부서의 관심사인 영역은 분리. 사내 맥락에서 특히 중요.

### 4순위: 변화 주기
자주 바뀌는 영역 vs 안정적인 영역 — 경계 후보. 마케팅 정책이 주간 단위로 바뀌면, 코어 거래 BC와 분리.

### 5순위: 언어 변화 (Ubiquitous Language)
같은 것을 설명하는데 어휘·추상화 수준이 달라지는 지점.

---

## 4. 진행 단계

### Step 1 — 1순위 증거 수집

UL 출력의 동음이의어 후보 전체 목록 → 각 후보가 어떤 속성 집합을 갖는지 재정리.

### Step 2 — 이벤트 흐름 클러스터링

이벤트 흐름에서 **함께 발생하고 함께 변하는** 이벤트 묶음을 찾는다.

예: [주문됨 → 결제완료됨 → 환불완료됨] 은 "주문·결제" 클러스터. [상품등록됨 → 상품노출됨] 은 "카탈로그" 클러스터.

### Step 3 — 조직·팀 경계 대조 (사내 맥락)

`project-kickoff`의 이해관계자·부서를 대조. 이미 조직이 나뉘어 있다면 BC도 그 경계를 존중하는 것이 관리 비용 면에서 유리.

### Step 4 — BC 후보 제안

각 후보에 대해:
- 이름 (업계 표준 어휘 우선)
- 포함 개념 (용어집에서 끌어옴)
- 핵심 책임
- 주요 이벤트
- **증거** (어느 기준 몇 순위에 따랐는지)

### Step 5 — 컨텍스트 간 관계 제안

BC 사이를 연결하는 관계 유형을 선택.

| 관계 | 의미 | 언제 쓰나 |
|------|------|-----------|
| **Shared Kernel** | 공유 모델/코드 | 드물게. 두 팀이 소유권을 공유할 때만 |
| **Customer-Supplier** | 상류가 하류의 요구 수용 | 서로 협업하며 우선순위 조율 |
| **Conformist** | 하류가 상류 모델을 그대로 따름 | 상류가 협상 불가(외부 시스템 등) |
| **ACL (Anti-Corruption Layer)** | 하류가 상류 모델을 변환해 격리 | 상류 모델이 우리 모델을 오염시킬 위험 |
| **Open Host Service** | 상류가 공용 프로토콜 공개 | 하류가 여러 개일 때 |
| **Published Language** | 공개된 스키마로 소통 | 프로토콜 문서화 + 버전 관리 |
| **Partnership** | 양 팀 함께 소유 | 공진화 필수 |
| **Separate Ways** | 통합하지 않음 | 이익보다 비용이 클 때 |

### Step 6 — 확인 및 출력

초안을 사용자에게 제시. 논쟁적 경계는 사용자 선택. 출력: `docs/context-map.md`.

---

## 5. 강제 질문 체크리스트

```
□ 동음이의어 후보 각각에 대해 BC 분리 여부 판단됨
□ BC 최소 2개 이상 제안됨 (단일 BC 시스템은 드뭄)
□ 각 BC에 "포함 개념"과 "핵심 책임"이 명시됨
□ 각 BC 경계에 "증거 (판별 기준 몇 순위)" 태그됨
□ BC 간 관계 최소 1종 이상 명시됨
□ 외부 시스템과의 관계(ACL / Conformist)가 명시됨
□ 사내 맥락이면 조직·팀 경계 대조됨
```

# Context Map

## Bounded Context 후보

### 1. 카탈로그 (Catalog Context)
- **포함 개념**: 상품(전시정보), 카테고리, 검색, 상세 설명
- **핵심 책임**: 상품 노출과 탐색
- **주요 이벤트**: 상품등록됨, 상품수정됨, 상품숨김처리됨
- **증거**:
  - 1순위: "상품"이 주문/배송과 다른 속성 집합 (UL 동음이의어)
  - 2순위: 이벤트 흐름의 상품 등록 클러스터

### 2. 주문 (Order Context)
- **포함 개념**: 주문, 주문상품 스냅샷, 장바구니
- **핵심 책임**: 구매 프로세스 조율
- **주요 이벤트**: 주문됨, 결제완료됨, 주문취소됨

### 3. 배송 (Shipping Context)
- **포함 개념**: 배송, 상품(무게·부피), 주소, 택배사
- **핵심 책임**: 물류 실행
- **주요 이벤트**: 출고요청됨, 발송됨, 배송완료됨

### 4. 정산 (Settlement Context) — 사내 맥락
- **포함 개념**: 판매자 정산, 수수료, 세금
- **핵심 책임**: 재무 정산
- **증거**: 3순위 (재무팀 소유)

## Context Map (관계)

```
┌──────────────┐        게시-구독       ┌──────────────┐
│   카탈로그   │──(상품정보 이벤트)──▶│    주문      │
└──────────────┘                        └──────┬───────┘
                                               │ 호출
                                               ▼
                                        ┌──────────────┐
                                        │    배송      │
                                        └──────────────┘
                                               ▲
                                               │ ACL
                                        ┌──────────────┐
                                        │ 외부 택배사  │(Conformist 상류)
                                        └──────────────┘
```

| From | To | 관계 | 비고 |
|------|------|------|------|
| 카탈로그 | 주문 | Customer-Supplier (게시-구독) | 상품 정보 변경 이벤트 전파 |
| 주문 | 배송 | ACL | 주문 도메인 어휘를 배송 어휘로 변환 |
| 배송 | 외부 택배사 | Conformist (우리가 하류) | 택배사 스펙 그대로 수용 |
| 정산 | 주문 | Customer-Supplier | 야간 배치로 데이터 수집 |

---

## ⚠️ 경계 판단이 애매한 지점

- **재고**가 카탈로그의 일부인지, 주문의 일부인지, 별도 BC인지 논쟁.
  - 증거 A: 카탈로그 이벤트와 함께 변함 → 카탈로그 귀속
  - 증거 B: 주문 성공률에 직접 영향 → 주문 귀속
  - 권고: 별도 **재고(Inventory) BC**로 분리 (Supporting 영역). 주문과 카탈로그 양쪽에서 조회.

---

→ 그 후 `aggregate-designer`로 각 BC 내부 Aggregate 설계.

## 7. 맥락별 조정

| 맥락 | BC 수 | 조직 경계 반영 | 비고 |
|------|-------|----------------|------|
| 학습 | 1~2개 | — | 단순하게 |
| 개인 | 2~3개 | — | 과도한 분리 금지 |
| 사내 | 3~6개 | 강함 | 조직 경계 존중 |
| B2B | 3~6개 | 중간 | 멀티테넌시 경계 별도 고려 |
| B2C | 3~7개 | 중간 | 업계 표준 관행 참고 |

## 8. 불확실성 표기

- 경계 판단이 50/50이면 **두 대안 모두 명시**하고 권고 이유를 붙인다.
- 외부 시스템과의 관계는 "가정"이 아니라 **실제 인터페이스를 확인**해야 함. 미확인 시 ⚠️.

## 10. 절대 하지 말 것

- 동음이의어 증거를 무시하고 기술 레이어(UI/Service/DB) 기준으로 BC 자르기.
- 모든 개념을 하나의 거대 BC에 몰아넣기.
- 외부 시스템 관계를 Partnership이나 Shared Kernel로 쉽게 잡기 (대부분은 ACL·Conformist).
- 경계를 확정했다고 선언하고 사용자 확인 없이 넘어가기.
- 컨텍스트 간 관계 생략.
