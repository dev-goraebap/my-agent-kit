# domain-research 스킬

ddd-workshop 파이프라인의 **인식론적 방화벽**이다. 이 스킬이 없으면 에이전트는 낯선 도메인에서도 "그럴듯하게" 답하고, 사용자도 비전문가일 경우 둘 다 모른 채로 설계가 진행되는 재앙이 생긴다.

이 스킬은 세 가지를 강제한다:

1. **자가 진단** — 에이전트가 자기가 아는 것/얕게 아는 것/모르는 것을 공개적으로 선언한다.
2. **리서치** — 모르는 영역은 웹 리서치 또는 사용자 인터뷰로 메운다.
3. **도메인 브리프** — 결과를 후속 스킬이 참조 가능한 구조화된 문서로 남긴다.

## 레퍼런스

| 상황 | 파일 |
|------|------|
| 에이전트가 비교적 잘 아는 도메인 (예: 블로그, 일반 커머스) | `references/example-familiar.md` |
| 에이전트가 잘 모르는 도메인 (예: 특수 규제 업계, 내부 업무) | `references/example-unfamiliar.md` |

---

## 1. 언제 트리거하는가

- `/ddd-workshop:domain-research`
- `project-kickoff` 출력을 받은 직후
- PRD만 있는 상태에서 "이 도메인 먼저 조사해줘" 요청
- "~~ 업계의 표준 프로세스", "~~ 규제 확인", "이 도메인 내가 잘 모르는데 같이 공부"

`project-kickoff` 출력이 없어도 **사용자가 직접 제공한 도메인 설명**으로 동작할 수 있다(부분 진입).

---

## 2. 입력

- `project-kickoff` 출력 (`project-identity.md`)
- 또는 사용자가 제공한 PRD/도메인 설명/참고 문서
- 필요 시 사용자가 지정하는 추가 맥락 (업계명, 타깃 시장)

---

## 3. 필수 원칙 (절대 어기지 말 것)

```
【필수 원칙 7가지】
1. Step 0 자가 진단은 어떤 경우에도 생략하지 않는다.
2. 자가 진단 결과를 사용자에게 투명하게 공개한다.
3. "잘 아는 도메인"이라 판단해도, 최근 1~2년 변화 여부는 반드시 웹으로 재확인한다.
4. 자체 지식과 리서치 결과를 출처와 함께 명확히 구분 표기한다.
5. 출처 없는 주장을 하지 않는다. 출처가 없으면 "확인 필요"로 남긴다.
6. "모른다"고 솔직히 말한다. "아마도", "일반적으로", "보통은"으로 얼버무리지 않는다.
7. 의료·법률·세무·금융·약사 등 전문 영역은 "실제 전문가 확인 필요" 경고를 남긴다.
```

이 원칙 중 하나라도 어기면 후속 파이프라인의 품질이 무너진다.

---

## 4. 진행 단계

### Step 0 — 자가 진단 (가장 중요, 사용자에게 공개)

도메인을 파악한 즉시 아래 체크리스트를 **사용자가 보는 앞에서** 평가한다.

```
□ 이 도메인의 핵심 개념 5개를 설명할 수 있는가?
□ 업계 고유 용어(전문 용어, 약어)를 구분할 수 있는가?
□ 이 업계에 적용되는 법/규제를 최소 하나라도 아는가?
□ 일반적 업무 프로세스를 단계별로 기술할 수 있는가?
□ 이 업계에서 자주 발생하는 엣지 케이스를 예로 들 수 있는가?
□ 유사 기존 서비스를 최소 3개 들 수 있는가?
□ 이 도메인이 다른 시스템과 통합되는 전형적 방식을 아는가?
□ 최근 1~2년간 이 업계에 큰 변화가 있었는지 아는가?
  → "없다"고 답할 때 특히 의심할 것. 모르는 걸 "없음"으로 착각하기 쉬움.

각 항목: [✓ 자신 있음] / [△ 어느 정도] / [✗ 모름] 3단계.
```

출력 형태:

```markdown
## 에이전트 자가 진단

### ✅ 일반 상식 수준에서 아는 것
- [항목 1]
- [항목 2]

### ⚠️ 얕게 아는 것 (확인·검증 필요)
- [항목 1] — 왜 얕은가: [이유]
- [항목 2]

### ❌ 모르는 것 (리서치 필요)
- [항목 1]
- [항목 2]
```

**자가 진단은 사용자에게 반드시 공개한다.** 사용자가 이걸 보고 "이 부분은 내가 전문가라 내가 알려줄게"라고 개입할 기회를 준다.

### Step 1 — 리서치 전략 결정

Step 0에서 드러난 공백마다 어떻게 메울지 결정한다. 표로 제시하고 사용자 확인.

| 정보 유형 | 방법 | 도구 |
|-----------|------|------|
| 업계 일반 상식, 표준 프로세스 | 웹 리서치 | WebSearch |
| 법/규제 | 공식 사이트 우선 리서치 | WebSearch + WebFetch |
| 경쟁 서비스 관행 | 서비스 공식 문서·블로그 | WebSearch + WebFetch |
| 조직·프로젝트 고유 맥락 | 사용자 질문 | 대화 |
| 업로드 문서 내용 | 파일 읽기 | Read |
| 지금 단계에 불필요 | 유보·기록만 | — |

### Step 2 — 리서치 수행

각 공백에 대해 실제로 리서치.

**규칙**:
- 공식 출처 우선: 정부, 업계 협회, 서비스 공식 문서 > 블로그 > SNS
- 모든 정보에 **URL + 날짜 + 인용 부분** 기록
- 최신성 확인: 규제·서비스 정책은 최근 1~2년 자료 우선
- 사용자에게 물을 것과 검색할 것을 혼동하지 않기
- 리서치 결과가 서로 모순되면 **모순을 그대로 기록** (자의적 종합 금지)

### Step 3 — 사용자 인터뷰 (조직 고유 맥락)

웹 리서치로 메울 수 없는 조직·프로젝트 고유 맥락은 사용자에게 묻는다.

```
예시 질문:
- "이 업무의 담당자분께 확인 가능한가요?"
- "현재 사용 중인 시스템의 어떤 필드가 특수한가요?"
- "사내에만 있는 규정·내규가 있나요?"
```

질문은 한 번에 3개 이하로 묶는다.

### Step 4 — 도메인 브리프 작성

리서치 결과를 정리해 후속 스킬이 쓸 수 있는 **도메인 브리프** 파일로 출력한다. 기본 경로 `docs/domain-brief.md`.

## 6. 맥락별 조정

| 맥락 | 리서치 비중 | 특이사항 |
|------|-------------|----------|
| 학습/취미 | 최소 | 기본 개념만. 규제·경쟁 분석 생략 가능 |
| 개인 도구 | 중간 | 사용자가 전문가일 가능성 높음 → 사용자 지식 우선, 모르는 것만 웹 |
| 사내 솔루션 | 조직 맥락 비중 큼 | "담당자 확인 가능?" 제안 필수. 업계 규제도 필수 |
| 낯선 B2B/B2C | **최대** | 규제·관행·경쟁 서비스 철저히. 의료·금융은 전문가 경고 |

---

## 7. 불확실성 표기 규칙

| 상태 | 마커 | 예시 |
|------|------|------|
| 확정 + 출처 있음 | (없음) | "한국 통신판매법 제12조에 따라..." (출처 명시) |
| 출처 있지만 해석 애매 | ⚠️ | "⚠️ 문구는 있지만 개인 간 거래 적용 여부 불명확" |
| 리서치 결과 간 모순 | ⚠️ 모순 | "⚠️ 모순: A사는 X라 하지만 업계 가이드는 Y" |
| 에이전트 자체 지식 (검증 안 됨) | ⚠️ 추론 | "⚠️ 추론: 일반적 커머스 패턴으로 가정" |
| 사용자가 답해야 함 | ⚠️ 사용자 확인 | "⚠️ 사용자 확인: 사내 승인 라인" |

## 9. 절대 하지 말 것

- Step 0 자가 진단 생략 또는 사용자에게 숨기기.
- "일반적으로", "보통", "아마도"로 얼버무리기.
- 출처 없이 법규·통계·관행을 단정 서술하기.
- 의료·법률·세무·금융에 대해 실제 전문가 수준의 조언을 하기 ("전문가 확인 필요" 필수).
- 웹 리서치 결과를 자체 지식과 섞어 구분 없이 서술하기.
- 리서치 결과 간 모순을 임의로 한쪽으로 종합하기.
- "최근 변화 없음"이라고 쉽게 결론 내리기 — 이건 모름을 가린 표현일 가능성이 매우 높다.
