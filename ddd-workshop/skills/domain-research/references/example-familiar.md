# 도메인 브리프: 개인 독서 기록 도구 (도서 메타데이터 도메인)

## 에이전트 자가 진단

### ✅ 일반 상식 수준에서 아는 것
- 책의 기본 식별자 (ISBN-10, ISBN-13)
- 주요 메타데이터 필드 (제목, 저자, 출판사, 출간일)
- 국내 도서 도메인에서 "번역서"와 "원서"의 구분
- 일반적 태그·독서 기록 앱의 기본 UX 패턴

### ⚠️ 얕게 아는 것
- 구글북스 API의 최신 무료 할당량·정책
  - 왜 얕은가: 학습 시점 이후 정책 변경 가능성
- ISBN 메타데이터 표준 (ONIX, MARC21)
  - 왜 얕은가: 개념은 알지만 필드 매핑 세부 미확인
- 알라딘 OpenAPI 가용성 및 조건

### ❌ 모르는 것
- 국립중앙도서관 서지정보 API(KOLIS-NET 등)의 현재 접근 방식
- 개인 저작권 관점에서 "발췌 문장 개인 저장"의 법적 경계
- 전자책 DRM과 개인 메모 연동 가능성

## 업계 일반 지식
- ISBN-13은 978 또는 979 접두사 + EAN 체크섬 구조.
  - 출처: [ISO 2108](https://www.isbn-international.org/content/what-isbn) — 2026-04-21 조회
- ONIX for Books는 국제 출판계 메타데이터 표준. 국내 출판사도 상당수 채택.
  - 출처: [EDItEUR ONIX](https://www.editeur.org/8/ONIX/) — 2026-04-21 조회

## 관련 법규 및 규제
- **저작권법 제28조 (공표된 저작물의 인용)**: 정당한 범위 안에서 공정한 관행에 합치되게 인용 가능.
  - 출처: [국가법령정보센터](https://www.law.go.kr/) — 2026-04-21 조회
  - ⚠️ "개인 메모용 발췌 보관"은 사적 이용 복제(제30조)에 포섭 가능성 높으나 공유 시 쟁점. 공유 기능 없는 개인용이므로 v1 범위에서 저위험.

## 업계 표준 프로세스
1. ISBN 입력/스캔 → 메타데이터 API 조회
2. 메타데이터 누락 시 수기 입력 백업
3. 사용자별 장서/독서 상태 관리 (읽음/읽는 중/읽고 싶음)
4. 발췌·메모 첨부
- 출처: 굿리즈, 밀리, 리디북스 등 UX 일반 패턴 관찰

## 주요 플레이어 및 관행
- **Goodreads**: 소셜 독서. 친구 공유 중심.
- **Notion 독서 템플릿**: 자유도 높지만 입력 마찰 큼.
- **밀리의 서재**: 구독형 전자책 + 독서 기록 결합.
  - 차별점: DRM 콘텐츠 내 하이라이트 저장이 앱 내부에 갇힘.

## 자주 발생하는 엣지 케이스
- 동일 책의 개정판·번역판이 각각 다른 ISBN → 같은 "독서 경험"으로 묶을지 분리할지 UX 결정 필요.
- 절판 도서는 메타데이터 API에서 누락 빈번 → 수기 입력 플로우 필수.
- ISBN이 없는 책 (자비 출판, 독립 출판) → 내부 식별자 체계 필요.

## ⚠️ 여전히 불확실한 것
- 구글북스 API의 국내 도서 커버리지 최신 실제 수치 (파일럿 테스트로 확인 필요).
- 알라딘 OpenAPI 승인 프로세스 최근 변화.

## ⚠️ 사용자에게 확인 필요한 것
- 주로 읽는 책의 장르·언어 비중 (메타데이터 소스 우선순위 결정에 영향).
- 발췌 문장의 평균 길이·빈도 (스토리지 설계 영향).

## 후속 스킬을 위한 힌트

### requirement-clarifier가 물어야 할 업계 특화 질문
- ISBN 없는 책을 어떻게 식별할지 (자비출판, 학술 자료, 논문)?
- 같은 책의 개정판을 별도 레코드로 둘지, 같은 Work로 묶을지?
- 발췌 인용 시 원문 페이지·위치 기록 방식?
- 독서 상태 전이 규칙 ("읽는 중" → "포기"도 허용?)

### ubiquitous-language-extractor가 주의할 용어
- "책" vs "작품(Work)" vs "판본(Edition)" — 도서 표준은 Work/Expression/Manifestation/Item 4계층(FRBR)을 쓴다. 사용자 언어는 단순 "책"일 것.

### event-flow-mapper가 누락하기 쉬운 이벤트
- 독서 시작 전 "읽고 싶음(Want-to-Read) 등록" 이벤트를 서비스 중심 패턴에서 빠뜨리기 쉬움.

### bounded-context-identifier가 참고할 구분선
- 도서 메타데이터 / 개인 독서 기록 / 발췌·메모 세 영역이 자연스럽게 분리됨 (언어도 달라짐: "제목" vs "상태" vs "문장").

### strategic-classifier가 참고할 경쟁 차별점
- Goodreads·밀리 모두 소셜 or 구독 중심. "장기 비공개 개인 자산화"는 공백 영역 → Core 후보.
