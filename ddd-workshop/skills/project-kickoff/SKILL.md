# project-kickoff 스킬

ddd-workshop 파이프라인의 **입구**다. 사용자가 던진 막연한 아이디어·PRD·러프한 프롬프트를 받아 **프로젝트 정체성**을 구조화된 문서로 확정하고, 다음 스킬(`domain-research`)에 넘길 수 있는 형태로 내보낸다.

"뭔지", "왜 만드는지", "누가 쓰는지", "어디까지 할 건지"를 대화로 확정하는 단계다. 여기서 **맥락을 잘못 판별하면 이후 파이프라인 전체가 어긋나므로** Step 0을 반드시 먼저 수행한다.

## 레퍼런스

`references/`에 맥락별 완성 출력 예제가 있다. 인터뷰에서 맥락이 정해지면 **해당 파일만 로드**해 톤·질문 깊이·출력 밀도를 맞춘다.

| 맥락 | 파일 |
|------|------|
| 학습/취미 | `references/example-learning.md` |
| 개인 도구 | `references/example-personal.md` |
| 사내 도구/솔루션 | `references/example-internal.md` |
| B2B | `references/example-b2b.md` |
| B2C 제품 | `references/example-b2c.md` |

---

## 1. 언제 트리거하는가

다음 상황에서 자동 트리거한다.

- "도메인 설계 시작해줘", "DDD 워크숍 시작", "/ddd-workshop:project-kickoff"
- "~~ 만들고 싶어", "~~ 서비스/시스템/플랫폼/툴 만들려고"
- "프로젝트 킥오프", "아이디어 정리해줘", "기획 초안 잡아줘"
- 사용자가 PRD 파일을 던지며 "여기서부터 도메인 설계로 이어갈래"

사용자가 이미 `lean-prd`로 기획서를 만든 상태라면 그 문서를 입력으로 받아 **프로젝트 정체성만 재정리**한다(중복 인터뷰 금지).

---

## 2. 입력

- 사용자의 자유 서술 (한 문장도 충분)
- 또는 기존 PRD/기획 문서 경로
- 또는 이전 대화에서 정리된 요구사항

**부분 진입 허용**: 입력이 풍부하면 공통 질문 일부를 생략할 수 있다. 단 맥락 판별(Step 0)은 절대 생략 금지.

---

## 3. 진행 단계

### Step 0 — 맥락 판별 (필수, 건너뛸 수 없음)

첫 발화에서 아래 표로 맥락을 추정한다.

| 발화 단서 | 추정 맥락 |
|----------|-----------|
| "연습", "공부", "배워보려고", "튜토리얼" | 학습/취미 |
| "내가 쓰려고", "나만 쓸", "개인용", "혼자 관리" | 개인 도구 |
| "우리 회사", "사내", "부서", "팀 업무", "우리 팀" | 사내 도구/솔루션 |
| "고객사", "납품", "B2B", "기업용" | B2B |
| "사용자 모으고", "런칭", "서비스 출시", "앱 스토어" | B2C 제품 |

추정이 서지 않거나 여러 맥락에 걸치면 **딱 한 번만** 확인:

```
"이 프로젝트는 어떤 성격인가요?
 (1) 학습/취미 — 배우거나 손맛으로
 (2) 개인 도구 — 내가 쓰려고
 (3) 사내 도구/솔루션 — 우리 팀·회사 내부
 (4) B2B — 외부 기업 고객에게 납품
 (5) B2C 제품 — 외부 일반 사용자에게 제공"
```

맥락 추정 신뢰도를 `confidence: {high|medium|low}`에 기록한다.

### Step 1 — 공통 필수 질문 (5가지, 맥락 무관)

맥락 판별이 끝나면 **한 번에 한두 개씩** 묻는다(Progressive Depth). 답변이 짧으면 한 번 더 파고든다.

1. **이 시스템은 무엇을 하는가?** — 한 문장으로 설명해달라.
2. **누가 사용하는가?** — 주요 액터/역할. (개인 도구라면 "나 혼자"도 답이 됨)
3. **어떤 문제를 풀거나 어떤 가치를 주는가?** — 이 질문이 Core Domain 판별의 근거가 된다.
4. **반드시 포함할 것 / 명시적으로 제외할 것은?** — "지금은 안 함"과 "영영 안 함"을 구분해서 받는다.
5. **기존 시스템이나 제약이 있는가?** — 기술 스택, 연동 시스템, 인력, 일정.

### Step 2 — 맥락별 추가 질문

#### 학습/취미
- 어디까지 만들면 "완성"으로 볼 건가?
- 이 프로젝트에서 배우고 싶은 기술·개념은?
- 시장 분석·지표는 **묻지 않는다**.

#### 개인 도구
- 내가 쓰기 시작하는 시점·상황은?
- 혼자 유지보수할 수 있는 복잡도 한계는?
- 페르소나·경쟁 분석은 **묻지 않는다**.

#### 사내 도구/솔루션
- 현재는 어떻게 일하고 있는가? (As-Is 프로세스)
- 연동할 기존 시스템이 있는가? (ERP, 사내 SSO, 기간계)
- 이해관계자 부서/담당자는? (이 질문이 나중에 Bounded Context 힌트)
- 규정·감사·컴플라이언스 요건이 있는가?
- 시장성·차별화는 **묻지 않는다**.

#### B2B
- 계약/도입 단위는? (기업당/좌석당/사용량)
- 고객사 간 요구 차이를 어떻게 수용하는가? (멀티테넌시, 커스터마이징)
- SLA·감사·보안 요구?

#### B2C 제품
- 타깃 사용자는 누구인가? (페르소나 1~2종)
- 유사 제품과의 차이는?
- 성공 지표는? (가입자, 거래액, 리텐션 등)

### Step 3 — 확인 및 출력

답변이 모이면 **사용자에게 초안을 보여주고 확인**받는다. 사용자가 수정/보완하면 반영 후 최종 출력.

---

## 4. 강제 질문 체크리스트

초안 작성 전 아래가 모두 채워졌는지 자체 점검한다. 빠진 게 있으면 **빠진 것만** 보충 질문.

```
□ 맥락 판별됨 (5종 중 하나)
□ 한 줄 정의
□ 해결하려는 문제 (Why)
□ 주요 액터
□ 반드시 포함 / 명시적 제외
□ 기술·일정·인력 제약
□ (사내) As-Is 프로세스
□ (사내) 연동 시스템
□ (사내) 규제·감사
□ (B2B) 계약 단위·멀티테넌시
□ (B2C) 타깃 사용자, 차별점, 성공 지표
```

"알아서 잘 채워주세요" 패턴 금지. 모르는 건 `⚠️ 미검증` 으로 남긴다.

## 5. 출력 포맷

출력 파일은 기본 `project-identity.md` (저장 위치는 사용자에게 묻는다. 기본 `docs/project-identity.md`).

```markdown
version: 0.1.0
context: {학습|개인|사내|B2B|B2C}

created_at: YYYY-MM-DD
---

## 개요
- **한 줄 정의**:
- **맥락**:
- **해결하려는 문제 (Why)**:
- **핵심 가치**:

- 주요 액터:
- (B2C) 타깃 사용자:

## 범위
- 반드시 포함:
- 명시적 제외 (Non-goals, 영영 안 함):
- 나중으로 미룸 (Later, 버전 힌트 선택):

## 제약 조건
- 기술:
- 일정·인력:
- (사내) 연동 시스템:
- (사내) 규제·감사:
- (B2B) 계약 단위·멀티테넌시:

```

### 출력 모드 옵션

사용자가 원하면 다른 포맷으로도 낼 수 있다.

| 모드 | 용도 |
|------|------|
| `raw` (기본) | 위 프로젝트 정체성 포맷 |
| `prd` | `lean-prd` 스킬이 이어받기 좋은 PRD 섹션 구조 |
| `canvas` | Lean Canvas 1페이지 (B2C 맥락에서 유용) |
| `domain-ready` | `domain-research` 입력에 최적화된 압축 포맷 |

## 6. 맥락별 조정 요약

| 맥락 | 질문 깊이 | 출력 밀도 | 주의 |
|------|-----------|-----------|------|
| 학습/취미 | 얕게 | 가볍게 | 시장·지표 질문 금지 |
| 개인 도구 | 중간 | 가볍게 | 페르소나·경쟁 질문 금지 |
| 사내 | 깊게 | 상세 | As-Is·이해관계자·규제 필수 |
| B2B | 깊게 | 상세 | 멀티테넌시·SLA 필수 |
| B2C | 깊게 | 상세 | 차별점·지표 필수 |

## 7. 불확실성 표기

추론한 내용, 사용자가 명시하지 않은 가정, 리서치가 필요한 사항은 **⚠️** 마커로 표시한다.

```markdown
- ⚠️ 추론: 사용자가 명시하지 않았지만, "지역 기반 신뢰"를 Why로 추론함 → 확인 필요
- ⚠️ 미확정: 결제 수단 범위
- ⚠️ 리서치 필요: 한국 중고거래 규제 (domain-research에서 처리)
```

이 마커는 이후 스킬들이 "아직 확정 안 됨"을 식별하는 신호다.

```
  에이전트 자가 진단 + 도메인 리서치를 수행해
  이후 요구사항 정제·Ubiquitous Language 추출의 근거가 되는

바로 요구사항 정제만 필요하다면:
→ `requirement-clarifier`를 호출할 수 있지만,
  업계 특화 질문이 약해지므로 domain-research를 권장합니다.
```

---

## 9. 절대 하지 말 것

- 맥락 판별(Step 0) 건너뛰기.
- 한 번에 5개 이상 질문 쏟아내기 (Progressive Depth 원칙 위반).
- 사용자가 명시하지 않은 내용을 확정 사실로 출력하기. 추론은 반드시 ⚠️.
- 학습/개인 맥락에 "타깃 사용자 페르소나"·"시장 분석" 묻기.
- B2C 맥락에 "사내 연동 시스템" 묻기.
- 출력에 `next_suggested` 메타데이터 빠뜨리기.
