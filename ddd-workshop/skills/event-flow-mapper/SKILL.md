# event-flow-mapper 스킬

Event Storming은 보통 포스트잇과 도메인 전문가 여럿이 벽 앞에 모여 하는 워크숍이다. 이 스킬은 그걸 **혼자 하는 축약판**으로 수행한다. 목적은 포스트잇 완벽주의가 아니라, **시간 흐름의 맥락**을 확보해 Bounded Context와 Aggregate 설계의 기반을 만드는 것.

핵심 결과물은 다음 세 가지:

1. **도메인 이벤트 흐름** (과거형, 시간 순)
2. **커맨드와 액터**
3. **정책(Policy)과 핫스팟**

## 3. 기본 용어 약속

이 스킬은 다음 5가지 요소를 구분한다 (Event Storming 표준).

| 요소 | 표기 | 예시 |
|------|------|------|
| 도메인 이벤트 | 과거형 | `주문됨`, `결제완료됨`, `재고복원됨` |
| 커맨드 | 능동형 | `주문하기`, `결제취소하기` |
| 액터 | 사람/외부 시스템 | `회원`, `관리자`, `결제 게이트웨이` |
| 정책 | 이벤트 → 커맨드 연결 규칙 | "결제완료됨 → 출고요청하기" |
| 핫스팟 | ⚠️ 논의 필요 지점 | "결제실패 시 재고 언제 복원?" |

**이벤트는 반드시 과거형으로** 쓴다. "주문 중" 같은 진행형은 이벤트가 아니다.

---

## 4. 진행 단계

### Step 1 — 액터 식별

먼저 시스템과 상호작용하는 **액터**를 나열한다. `project-kickoff`의 액터 목록에서 가져오되, **외부 시스템도 액터**임을 잊지 않는다 (결제 게이트웨이, 이메일 서버, SAP 등).

### Step 2 — 시드 이벤트 추출

요구사항·용어집에서 "이런 일이 일어난다"는 표현을 찾아 **과거형 이벤트**로 변환.

- "회원이 주문을 넣는다" → `주문됨`
- "결제가 완료되면 알림을 보낸다" → `결제완료됨`, `알림전송됨`
- "재고가 부족하면 주문이 실패한다" → `주문실패됨` (원인: 재고부족)

이벤트 후보를 **시간 순**으로 줄 세운다. 순서가 모호하면 ⚠️ 핫스팟.

### Step 3 — 커맨드와 액터 연결

각 이벤트 앞에 "누가 무엇을 시켰는가"를 붙인다.

```
[회원] →(주문하기)→ [주문됨]
[결제 게이트웨이] →(결제통지)→ [결제완료됨]
```

액터 없이 시간만으로 발생하는 이벤트도 있음(스케줄러, 배치, 타임아웃).

### Step 4 — 정책 추출

이벤트 → (자동으로) → 커맨드 연결을 **정책(Policy)** 로 명명.

```
[주문됨] ─정책:재고확인─→ (재고확인하기) → [재고확인완료됨]
[결제완료됨] ─정책:출고요청─→ (출고요청하기) → [출고요청됨]
```

정책은 나중에 `aggregate-designer`에서 **Specification / Domain Service / Event Handler** 중 어디에 둘지 결정된다. 지금은 "반응 규칙이 있다"까지만.

### Step 5 — 분기와 핫스팟

동일 커맨드가 여러 결과를 낼 수 있는 지점을 표시.

```
(결제하기) → [결제완료됨]  or  [결제실패됨]  or  [결제타임아웃됨]
```

의사결정이 모호한 지점은 ⚠️ 핫스팟으로 명시. 이 지점이 곧 `requirement-clarifier` 재호출이나 사용자 확인 후보.

### Step 6 — 업계 표준 대비 누락 점검

`domain-brief`의 "업계 표준 프로세스" 단계와 현재 이벤트 흐름을 대조.

예 (제조 QMS):
- 업계 표준: 부적합 발견 → **격리(Quarantine)** → 근본원인분석 → CAPA
- 현재 흐름: 부적합 발견 → CAPA
- → ⚠️ "격리" 단계 누락 가능성. 사용자 확인 필요.

### Step 7 — 확인 및 출력

초안을 사용자에게 보이고 검증 후 출력. 기본 경로 `docs/event-flow.md`.

# 이벤트 흐름 맵

## 식별된 액터
- **회원** — 주문·결제 주체
- **관리자** — 상품·정산 관리
- **결제 게이트웨이** (외부)
- **스케줄러** (내부, 자동 트리거)

## 주요 플로우

### Flow A: 주문·결제·출고

```
[회원]
  │ (주문하기)
  ▼
[주문됨]
  │ 정책 P1: 재고 확인
  ▼
(재고확인하기) ──▶ [재고확인완료됨] ──or──▶ [재고부족됨]
                        │
                        │ 정책 P2: 결제 요청
                        ▼
                  (결제요청하기)
                        │
                        ▼
                [결제완료됨] ──or──▶ [결제실패됨]
                        │
                        │ 정책 P3: 출고 요청
                        ▼
                 [출고요청됨]
```

### Flow B: 주문 취소·환불

```
[회원] →(주문취소하기)→ [주문취소됨]
                          │ 정책 P4: 결제 역전
                          ▼
                   [환불완료됨] or [환불보류됨]
```

### Flow C: 배치·스케줄

```
[스케줄러, 매일 02:00] →(미결주문정리배치)→ [타임아웃주문취소됨]
```

---

## 식별된 정책

| ID | 이름 | 트리거 이벤트 | 유발 커맨드 | 비고 |
|----|------|----------------|-------------|------|
| P1 | 재고 확인 | 주문됨 | 재고확인하기 | 실시간 |
| P2 | 결제 요청 | 재고확인완료됨 | 결제요청하기 | |
| P3 | 출고 요청 | 결제완료됨 | 출고요청하기 | 외부 WMS 연동 |
| P4 | 결제 역전 | 주문취소됨 | 환불요청하기 | 결제 상태 따라 조건부 |

---

## 핫스팟 ⚠️

- ⚠️ **결제 실패 시 재고는 즉시 복원? 타임아웃까지 예약?**
  - 즉시 복원 시 동시성 이슈, 예약 시 과매도 리스크.
  - 사용자 결정 필요.
- ⚠️ **부분 결제 허용?**
  - 여러 결제수단 조합 허용 여부.
- ⚠️ **환불 보류 조건이 불명확** — 어떤 경우에 보류로 갈지.

---

## ⚠️ 업계 표준 대비 누락 가능성

- `domain-brief` § 업계 표준 프로세스에 따르면, [분쟁 조정] 단계가 일반적이나 현재 흐름에 없음.
- `domain-brief` § ...에 따르면, [격리] 단계가 부적합 이후 반드시 오지만 현재 흐름에 없음.

- ⚠️ 핫스팟은 반드시 별도 섹션으로 분리.

- 정책이 "자동으로 발생하는지"와 "사람이 승인해야 하는지" 구분 — 불명확하면 ⚠️.

## 10. 절대 하지 말 것

- 이벤트를 현재형·진행형으로 쓰기 ("주문 중" → 이건 이벤트가 아님).
- 외부 시스템 액터 빠뜨리기.
- 비정상 경로(실패·타임아웃·취소) 없이 "해피 패스"만 그리기.
- 핫스팟을 발견하고도 "문제 없음"으로 적기.
- 정책을 전부 이벤트 핸들러로 뭉뚱그리기 (구현 결정은 aggregate-designer의 몫).
- 용어집에 없는 새 어휘를 임의로 도입하기 — UL 일관성 깨짐.
